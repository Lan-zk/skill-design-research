# Agent Skill 架构设计与模型能力解耦规范

## 1. 规范目标
规范 Skill 的开发边界，解决独立脚本重复调用外部模型接口导致的 API Key 依赖问题，实现宿主模型原生推理能力与底层执行逻辑的彻底解耦。

## 2. 核心分层原则
* **非确定性任务上浮**：所有涉及自然语言理解、逻辑推理、分析、总结、翻译的非确定性任务，必须由宿主 Agent (Host Agent) 或其委派的子智能体 (Subagents) 承担，禁止将此类逻辑下沉封装至底层执行脚本中。
* **确定性任务下沉**：底层执行脚本（如运行在 MacBook Air M4 本地终端或沙箱中的 Python/Bash 脚本）仅负责文件读写、格式校验、正则匹配、确定性 API 数据交互等强确定性任务。
* **零外部 API Key 依赖**：Skill 目录的 `scripts/` 文件中，严禁硬编码大模型请求逻辑，严禁在 Skill 运行时强制要求环境提供额外的模型调用凭证（API Key）。

## 3. 标准化工作流模式 (Workflow Patterns)

在需要“翻译”、“总结”或“分析”时，必须采用以下标准架构模式之一替代脚本内发起的 API 请求：

### 模式 A：流水线编排模式 (Pipeline Pattern)
将模型能力定义为工作流中的标准节点。
1. **阶段 1 (分析/推理)**：宿主 Agent 自身读取目标文件内容，依据 `references/` 中的规范标准，利用自身上下文窗口完成分析或翻译计算。
2. **阶段 2 (执行/固化)**：宿主 Agent 将经过自身推理生成的结构化纯文本结果，作为参数传递给 `scripts/` 下的确定性脚本，完成最终的写入、排版或系统交互。

### 模式 B：子智能体委派模式 (Subagent Delegation)
针对复杂且可能污染主上下文的分析需求，利用框架原生机制拉起分身。
1. **阶段 1 (委派)**：宿主 Agent (Orchestrator) 通过框架级别的工具（如 `TaskTool`），派生出具有独立运行上下文的专家子智能体。
2. **阶段 2 (并行处理)**：子智能体共享框架底层的统一模型调用通道和鉴权状态，完成复杂的总结分析。
3. **阶段 3 (合成)**：主智能体接收子智能体返回的标准化分析摘要，并进入下一步工作流。

## 4. 架构审查清单 (Architecture Review Checklist)
在 Skill 打包或合并前，必须通过以下自检：
- [ ] Skill 的执行脚本流中，是否彻底移除了对额外大模型 API Key 的依赖？
- [ ] `scripts/` 目录下的代码逻辑中，是否彻底剥离了 Prompt 构建逻辑？（Prompt 应固化在 `SKILL.md` 或 `references/` 中）
- [ ] 是否存在“将非结构化文本交给 Python 脚本去提取大意”的反模式？（若有，需上浮交由 Agent 处理）
