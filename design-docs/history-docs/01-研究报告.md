# Skills 设计研究报告

## 1. 研究目标

本报告回答四个核心问题：

1. 为什么有些 skill 看起来只有一个入口，却能覆盖很多能力？
2. skill 应该按最小能力单元切分，还是按完整工作流切分？
3. 顶层 skill 数量为什么不宜过多？
4. 一个“好 skill”的设计边界、内部结构、评估方法，应该长什么样？

本次研究不是要优化样本仓库本身，而是把它们当作样本，抽象出可复用的方法论。

## 2. 研究样本

### 2.1 本地样本

#### 样本 A: `ui-ux-pro-max-skill`

关键观察：

- 对外暴露的是一个顶层技能入口，而不是把全部子能力都直接暴露给模型。
- 实际能力来自多层结构：
  - `SKILL.md` 负责触发与工作流
  - `src/ui-ux-pro-max/data/` 负责结构化知识
  - `scripts/search.py`、`core.py`、`design_system.py` 负责可执行检索与组合逻辑
  - `templates/` 负责安装与分发模板
- 仓库中同时存在 `.claude/skills/`、`cli/assets/`、`src/ui-ux-pro-max/` 三层，但 `src/ui-ux-pro-max/` 被明确为 source of truth。

这说明它不是“一个 prompt 很长”，而是“一个顶层 skill + 一个数据和脚本驱动的设计系统引擎”。

#### 样本 B: `skill-creator`

关键观察：

- 它不是一个业务 skill，而是一个“构建 skill 的元 skill”。
- 它定义的不是某个领域知识，而是完整的 skill 生命周期：
  - 明确意图
  - 编写 draft
  - 设计测试 prompt
  - 做 with-skill / without-skill 对比
  - 设计 assertions
  - 运行 grader / analyzer / comparator
  - 优化 description
  - package skill
- 它把 skill 设计从“写一个 `SKILL.md`”升级成了“规范 + 测试 + benchmark + packaging”的工程流程。

这说明一个成熟 skill 的设计，不能只看 prompt，还要看它是否定义了评估与迭代闭环。

#### 样本 C: `Agent-Skill-五种设计模式.md`

该文将 skill 设计抽象为五种模式：

- Tool Wrapper
- Generator
- Reviewer
- Inversion
- Pipeline

这五种模式非常有价值，但更重要的发现是：

**它们不是互斥分类，而是可组合的设计积木。**

一个成熟的 skill 往往不是单一模式，而是复合模式。

### 2.2 外部资料

#### Anthropic Agent Skills 文档

Anthropic 官方文档明确给出三层加载机制：

- Level 1: metadata，总是加载
- Level 2: `SKILL.md`，触发后加载
- Level 3+: resources / code，按需加载

这直接证明了 progressive disclosure 不是技巧，而是 Agent Skills 的基础架构。  
来源：[Agent Skills Overview](https://docs.claude.com/en/docs/agents-and-tools/agent-skills)

Anthropic 最佳实践又进一步强调：

- `SKILL.md` 要简洁
- description 是关键触发面
- `SKILL.md` 最好控制在 500 行内
- 复杂内容应拆到 references
- description 既要说“做什么”，也要说“什么时候用”

来源：[Skill authoring best practices](https://docs.claude.com/en/docs/agents-and-tools/agent-skills/best-practices)

#### OpenAI function calling 指南

OpenAI 官方建议：

- 初始可用函数数量保持较少
- 推荐在一个 turn 开始时少于 20 个函数，这只是软建议
- 如果函数太多，应使用 tool search 或延迟加载
- 函数定义本身会占用上下文

来源：[Function calling guide](https://developers.openai.com/api/docs/guides/function-calling)

这个结论虽不是直接讨论 skill，但它揭示了同一类问题：

**当模型面对过多扁平可选能力时，选择质量和上下文效率都会下降。**

#### Microsoft Research: Tool-space interference

Microsoft Research 给出了更直接的论证：

- 工具空间越大，越容易出现选择冲突、状态错位、命名歧义、参数复杂度过高、响应过长等问题
- 文章直接引用并认可“函数数量越多，性能越差”的趋势
- 文中还提到 flatten parameter space 这类简化设计可以显著提升工具调用表现

来源：[Tool-space interference in the MCP era](https://www.microsoft.com/en-us/research/blog/tool-space-interference-in-the-mcp-era-designing-for-agent-compatibility-at-scale/)

#### Google ADK workflow / multi-agent 文档

Google ADK 文档给出一套很成熟的 workflow primitives：

- SequentialAgent
- ParallelAgent
- LoopAgent
- Generator-Critic 模式
- Fan-out / Gather 模式

来源：

- [Workflow / Multi-agent documentation](https://google.github.io/adk-docs/agents/multi-agents/)
- [Tool performance / parallel execution](https://google.github.io/adk-docs/tools-custom/performance/)

这说明一个复杂 agent 系统的最佳实践，不是把所有能力平铺给主模型，而是让顶层 orchestrator 编排子工作流。

#### ToolNet 论文

ToolNet 论文指出：

- 传统做法把大量工具直接平铺成文本描述，要求 LLM 自己完成全部检索和决策
- 这种方式在大规模工具库下很快失效
- 论文提出 tool graph，把“工具检索”从平铺列表变成结构化导航

来源：[ToolNet: Connecting Large Language Models with Massive Tools via Tool Graph](https://arxiv.org/abs/2403.00839)

这为 skill 设计提供了很重要的研究支撑：

**大规模能力不应以平铺暴露的形式呈现给模型，而应以分层、检索、路由的形式呈现。**

## 3. 关键发现

## 3.1 一个 skill 能很强，不是因为 prompt 很长，而是因为它把能力外置了

`ui-ux-pro-max-skill` 的强，不来自一个超长 `SKILL.md`，而来自四件事：

1. `SKILL.md` 明确路由范围
2. `search.py` / `design_system.py` 把复杂判断变成可执行逻辑
3. CSV 数据库把大量设计知识变成可检索结构化资产
4. 安装模板把跨平台分发和落地做成了工程化流程

所以它看起来是“一个 skill 很强”，本质上却是：

**一个顶层工作流入口 + 一组外置能力层。**

## 3.2 `skill-creator` 说明：skill 的真正单位不是 prompt，而是产品化工作流

`skill-creator` 最重要的贡献不是“会写 skill”，而是它把下列内容纳入了 skill 规范：

- 描述如何触发
- 描述如何测试
- 描述如何比较基线
- 描述如何做人工评审
- 描述如何做 blind comparison
- 描述如何打包交付

它说明一个成熟 skill 至少有四个层次：

- 触发层
- 工作流层
- 评估层
- 发布层

## 3.3 “3 到 4 个，不超过 7 个”的说法更像经验法则，不是硬规范

研究结论如下：

- 我没有找到任何一家官方直接规定“最好只有 3 到 4 个，不超过 7 个 skills”
- 但我找到了多条强相关证据：
  - OpenAI 建议一个 turn 初始可用函数尽量少，软建议少于 20
  - Microsoft 明确讨论 large tool spaces 带来的 interference
  - Anthropic 强调 description 触发和按需加载，天然是在避免把所有内容提前暴露
  - ToolNet 则从研究上证明，大规模工具要靠结构化导航，而不是平铺

因此可以得出一个**推导性结论**：

> 顶层可见 skill 数量应该少，默认 3 到 4 个最稳，5 到 7 个仍可接受，但需要强边界。超过 7 个时，必须有明显的路由层或分组层。

这不是官方硬性限制，而是基于多份资料推导出的工程建议。

## 3.4 skill 的边界不应该按最小原子能力切

很多人第一次设计 skill，会倾向于这样切：

- 配色 skill
- 字体 skill
- layout skill
- animation skill
- accessibility skill

这在知识管理上看似细，但对模型路由并不友好，因为用户很少按这种粒度表达需求。

用户更常说的是：

- 帮我设计 landing page
- 帮我做设计系统
- 帮我 review UI
- 帮我把品牌规范沉淀成文档

也就是说，**用户表达的是工作流，不是知识原子**。

因此更合理的切法是：

- 顶层按工作流 skill 切
- 底层按原子能力组织资源

这是本次研究最重要的结论之一。

## 4. 对用户问题的直接回答

### 问题 1：skills 应该按最小单元能力设计，还是按完整工作流设计？

结论：

- **对外暴露给模型选择的 skill，应该按完整工作流设计**
- **对内沉淀的 references / scripts / templates / data，应该按原子能力组织**

也就是：

**外层按工作流，内层按原子能力。**

### 问题 2：为什么 `ui-ux-pro-max-skill` 这样一个 skill 可以支持很多能力？

因为它对外看是一个 skill，对内其实是一个能力系统：

- 一个广泛但聚焦的触发面
- 一个稳定的主工作流
- 一个结构化知识库
- 一个可执行搜索和推理引擎
- 一个设计系统生成器
- 一套跨平台分发模板

它不是“一个 skill 做很多不相关的事”，而是“一个 skill 做一个大但统一的工作流”。

### 问题 3：`skill-creator` 怎么定义 skill 的构建规范？

它通过以下方式定义规范：

- 规定 frontmatter 和 description 的重要性
- 规定 progressive disclosure 的结构
- 规定测试、benchmark、grading、comparison 的流程
- 规定 packaging 和 schema
- 把 skill 从“文本说明”升级为“工程对象”

### 问题 4：一个好的 skill 的基础能力、边界能力应该如何设计？

结论：

- 基础能力：完成一个明确用户任务所需的最小闭环能力
- 边界能力：处理相邻场景、异常、评审、回退和组合
- 最佳状态：不是能力越多越好，而是“一个用户任务闭环越完整越好”

## 5. 最终研究结论

一个好的 skill，不应被设计成下面两种极端：

- 极端 A：一个 skill 只有一个非常细的原子能力
- 极端 B：一个 skill 什么都做

最佳设计是中间态：

- 顶层 skill 数量少
- 每个 skill 对应一个清晰工作流
- skill 内部通过 references / scripts / templates / data 扩展深度
- 复杂任务通过 workflow 或多 skill 编排完成
- 评估与基线比较是 skill 设计的一部分，不是后补工作
