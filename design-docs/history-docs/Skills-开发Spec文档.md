# Agent Skills 开发与设计规范 (Development Spec)

本文档基于一系列 Skills 核心设计理念、架构与设计模式研究报告提炼而成，旨在为 Agent Skills 的开发、组织与评估提供标准化的工程规范与指导。

## 1. 核心理念与设计原则

- **Skill 不是 Prompt，而是工作流**：Skill 应被理解为“可被模型路由的工作流能力单元”。一个成熟的 Skill 必须包含四个维度：**触发维度**（何时调用）、**工作流维度**（如何完成任务）、**输出维度**（交付什么结果）和**评估维度**（如何衡量成功）。
- **外层按工作流，内层按原子能力**：不要按“知识点”（如字体、配色）来切分顶层 Skill，而应按“用户意图和任务闭环”（如设计系统、页面构建）来组织。顶层暴露少量工作流入口，底层沉淀深层的模块化能力。
- **顶层可见 Skill 数量少而清楚**：默认推荐 3 到 4 个顶层 Skill，谨慎上限为 7 个。超过此数量应引入分组或路由层，以避免工具空间干扰（Tool-space interference）和上下文浪费。

## 2. 目录与分层架构规范

一个标准的、非平凡（non-trivial）的 Skill 仓库应遵循渐进式披露（Progressive Disclosure）的目录结构：

```text
my-skill/
|-- SKILL.md         # 必选：工作流外壳与路由面
|-- references/      # 推荐：长文本知识、规范与审核 checklist
|-- scripts/         # 推荐：可执行校验、检索与转换逻辑
|-- templates/       # 可选：输出模板、HTML/Markdown 壳子 (或 assets/)
|-- data/            # 可选：结构化数据库（CSV/JSON、评分规则等）
`-- evals/           # 推荐：评估基准与测试用例
```

### 各层职责界定
- **`SKILL.md`**：只负责任务定义、触发边界、工作流步骤和资源导航。**绝不存放**大段背景知识、具体样例或结构化数据。
- **`references/`**：承载按领域切分的按需加载知识（如风格指南、规范文档）。
- **`scripts/`**：将重复性、易错、高风险的确定性步骤（如打包、校验、检索）外移成脚本。参数设计应尽量平坦。
- **`templates/` / `assets/`**：确保输出结构的一致性，提供起步文件（starter artifacts）。
- **`data/`**：承载分类学、评分规则、配置表等结构化数据。

## 3. SKILL.md 编写规范

### 3.1 Frontmatter 规范
- **`name`**：小写、连字符分隔，使用动名词或动作导向名词（如 `processing-pdfs`, `reviewing-ui`），不要泛化成 `helper` 或 `utils`。
- **`description`**：这是**第一触发面**。必须同时说明“**这个 skill 做什么**”以及“**什么时候应该触发它**”。必须清晰、聚焦，不是单纯的营销口号。

### 3.2 边界判定（新建/拆分测试）
只有同时满足以下四项，才建议拆分为新的顶层 Skill：
1. 触发语言明显不同
2. 工作流明显不同
3. 输出物明显不同
4. 评估标准明显不同

*如果只是数据不同、参考资料不同或脚本不同，不应拆分。*

## 4. 五大核心设计模式

在组织 Skill 内部指令和工作流时，推荐采用或组合以下五种设计模式：

1. **工具封装 (Tool Wrapper)**：按需加载特定库或框架的上下文。Agent 仅在触发时从 `references/` 加载相关开发规范并强制执行，避免污染全局提示词。
2. **生成器 (Generator)**：强制输出的一致性。通过加载 `templates/` 和 `references/`，向用户收集缺失的变量，然后填充并生成结构化文档。
3. **审查器 (Reviewer)**：分离“检查什么”与“如何检查”。Agent 动态加载外部 `review-checklist.md`，逐条应用规则，并按严重程度（Error/Warning/Info）生成结构化审计报告。
4. **反转 (Inversion)**：颠覆“用户提示-Agent执行”的直觉，让 Agent 扮演采访者。通过门控指令（如“在所有阶段完成前不得开始构建”），强制 Agent 结构化提问并收集完整需求与约束后再行动。
5. **流水线 (Pipeline)**：适用于高风险复杂任务。强制执行带有硬性检查点（Checkpoints）的多步骤工作流，明确规定必须由用户确认或校验通过后，才能进入下一阶段。

**模式组合原则**：复杂 Skill 默认是多模式的组合。例如处理模糊需求：`Inversion` (需求收集) -> `Pipeline` (分步执行) -> `Reviewer` (成果检查)。

## 5. 整体系统架构推荐

对于中大型系统或多 Agent 协作场景，推荐采用**默认四技能架构**：
- **`plan`**：需求收集、澄清、切分范围、产出方案。
- **`build`**：主体生成、执行与落地实现。
- **`review`**：质量审查、基准对比、验证与反馈。
- **`meta`**：Skill 自身的维护、打包、评测与演进。

对于目标高度集中的场景，可采用**单技能架构**（如专门的 UI/UX 设计系统），但必须保证底层资源（Scripts/Data/Templates）深度模块化，做到“统一输出物闭环”。

## 6. 评估与持续迭代规范 (Eval)

非平凡的 Skill 必须从第一天起就具备可评估性，这是衡量其成熟度的标准：
- **触发 Eval**：需包含 8-10 条 `should-trigger` 测试和 8-10 条 `should-not-trigger` 测试，必须包含易混淆的边界情况（near-miss）。
- **工作流 Eval**：测试用例必须覆盖 Happy path、Ambiguous path（模糊输入路径）和 Failure/Edge path。
- **Baseline 比较**：新技能开发需对比 `with-skill` vs `without-skill`；迭代优化需对比 `new-skill` vs `old-skill`（比较通过率、Token 消耗、耗时、失败模式等）。
- **断言 (Assertion)**：设计实质性的断言。判断“输出是否通过了验证脚本”、“是否满足结构约束”、“是否真正解决了业务需求”，而非简单的“文件是否存在”。
