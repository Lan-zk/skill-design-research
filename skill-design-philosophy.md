# Skill 设计哲学

> 设计/开发 agent skill 时的参考手册——给标准、给方案选项、给实现思路。内容从多个开源 skill 仓库的实践中提炼，但本文档独立可用，不依赖任何特定项目上下文。
>
> 用法：先读 §2 诊断你的不信任点 → §3 选型 → §5 查模式 → §7 落地 → §4 自检 + §6 兜底。

## 1. 基础契约（Agent Skills 规范）

Agent Skills 是 Anthropic 发起的开放标准（agentskills.io），定义了 skill 的可加载格式。

### 最小契约

一个 skill = 一个目录 + 目录里的 `SKILL.md`（YAML frontmatter + Markdown 正文）。目录名即技能名。可选 `scripts/`（可执行代码）、`references/`（补充文档）、`assets/`（模板/资源）。

### frontmatter 字段

| 字段 | 必填 | 约束 |
|---|---|---|
| `name` | 是 | 1-64 字符；小写字母/数字/连字符；不得开头结尾或连续连字符；**必须与父目录名一致** |
| `description` | 是 | 1-1024 字符，非空；应说"做什么 + 何时用"+触发关键词 |
| `license` | 否 | 许可证名或指向 LICENSE 文件 |
| `compatibility` | 否 | ≤500 字符；运行环境需求 |
| `metadata` | 否 | string→string 任意键值（扩展属性的合法位置） |
| `allowed-tools` | 否 | 空格分隔的预批准工具（实验性） |

只这 6 个是规范允许的 top-level 字段；多出来的会被验证器标 "Unexpected fields"——provider 扩展应塞进 `metadata`。

### Must / Recommended / Convention 三分

| 类别 | 含义 | 例子 |
|---|---|---|
| **Must** | 规范硬性规定，验证器强制 | 目录+SKILL.md；name/description 必填及字符规则；6 字段白名单 |
| **Recommended** | 建议，验证器不查 | 500 行/5000 token 上限；三层渐进式披露；description 含关键词 |
| **Convention** | de facto 约定（实践中普遍使用，未强制） | `.agents/skills/` 路径；project 覆盖 user；`disable-model-invocation` 过滤标志 |

**两条易误读边界**：
- 500 行是**建议不是硬规则**。多个生产级 skill 超 500 行且合理（信息密度高时自包含比拆引用更优）。
- `disable-model-invocation` 被实现指南承认为过滤标志，但**未进规范白名单**——属 provider 扩展，跨客户端兼容性会受影响。

## 2. 第一性原理：诊断决定药方

skill 设计的根决策不是"用什么模式"，而是"你认为 agent 出错主要出在哪一步"。诊断决定药方——不是"先进 vs 落后"。

### 四类不信任 → 四种药方

| 不信任点 | 药方 | 代表做法 | 失效条件 |
|---|---|---|---|
| **理解力** | 解释为什么，不堆规则 | frontmatter 极简，靠自然语言讲清"为什么"；脚本黑盒化 | 模型理解力弱时"解释"是噪声；不可信环境下不限工具是漏洞 |
| **知识** | 把知识外化成文件让模型读 | 对抗过时知识的清单；CSV+检索；官方规范镜像 | 知识频繁变动时维护成本高；隐式场景触发不到 |
| **纪律** | 流程/红线逼它 | 强制引导（session-start hook）；红旗表堵借口；路由器分流 | 引导是软约束，弱模型拦不住；用户一喊就放 |
| **审美** | 代码硬拦 | 确定性规则引擎；版式验证器；红线列表锁下限 | 主观品味代码检测不准；约束太死失去风格 |

多数 skill 主导一类但沾几类。药方跟着诊断走。

**四问定位**：你的 agent 出错是——
- "不理解为什么" → 理解力（解释）
- "不知道" → 知识（外化）
- "知道但不做" → 纪律（逼流程）
- "做了但跑偏" → 审美（硬拦）

## 3. 设计决策

每个决策给选项 + when-to-apply。不是选"最先进"，是选"匹配你的诊断"。

### 3.1 知识形态 → 披露策略

| 知识形态 | 披露策略 | when-to-apply |
|---|---|---|
| 散文原则（方法论、判断） | 单文件自包含 | 成型稳定、不能表格化 |
| 结构化数据（产品类型→配置） | 数据库 + 检索引擎 | 可枚举、易变、需查询 |
| 确定性规则（反模式、版式） | 规则引擎/验证器 | 可代码检测的通病 |
| 官方规范（格式、API） | 镜像进技能 | 有权威外部源 |

判据三问：能表格化吗？稳定规则还是易变数据？需要脚本执行吗？

### 3.2 触发方式

| 方式 | when-to-apply |
|---|---|
| 强制引导（session-start hook） | 技能不会被主动用除非强制；harness 支持 hook |
| description 匹配（默认） | 依赖模型自动触发 |
| 收窄模型自主权（disable-model-invocation） | 有些技能绝不该自动触发 |
| 路由器分发 | 顶层技能数 ≥ 5-7 |

模型有"未充分触发"倾向——补偿三档：强势 description（关键词枚举 + 何时用/不用）/ 路由器 / 强制引导。

### 3.3 可靠性执行：说服 vs 机械拦截

| 派别 | 手段 | when-to-apply |
|---|---|---|
| 说服 | prose 原则、红线列表 | 失败是"模型没意识到" |
| 机械 | 代码检测、验证器、CI | 失败是"知道但控制不住手"；失败可代码检测 |
| 混合 | 规则拦客观点 + LLM 批判兜底主观点 | 客观与主观失败并存 |

### 3.4 渐进式披露结构

| 结构 | when-to-apply |
|---|---|
| 小入口 + references/ | 内容量大且偏参考性 |
| 大自包含 | 信息密度高、跳转更亏 |
| 脚本化数据库 | 核心价值是检索 |
| sections/manifest 被动注册 | 超长工作流按需读 section |
| 强制引用加载 | 子命令引用必读（跳过就产出通用输出） |

500 行建议 vs 突破：信息密度高、大部分任务都需要大部分内容时自包含更优；否则拆引用。

### 3.5 frontmatter 哲学

| 取向 | when-to-apply |
|---|---|
| 极简（name+description） | 信任模型理解力 |
| 丰富（allowed-tools/version/hooks） | 收敛工具面、分级加载 |
| 调用轴划分（disable-model-invocation/user-invocable） | 划分谁触发 |

注意：`disable-model-invocation`/`user-invocable`/`version`/`argument-hint` 都不在规范白名单——provider 扩展，跨客户端兼容性受影响。

### 3.6 跨平台分发

| 方式 | when-to-apply |
|---|---|
| 不放集成文件靠规范 | 单一标准渠道 |
| 少量手写 | 1-3 个平台 |
| 手写镜像（占位符替换） | 平台多、差异小 |
| 模板引擎生成 | 平台多（≥5）、差异结构化 |
| 代码生成 host 配置 | 平台多且有 host 特定逻辑 |

**单源 + 镜像的同步是固有脆弱点**——只要有多份拷贝，必须配 CI/sync 校验，否则漂移不可避免。零拷贝（单文件即源）最稳但锁单平台。

### 3.7 评估与测试

| 方式 | when-to-apply |
|---|---|
| TDD 压力测试先行 | 失败成本高、有工程能力 |
| evals baseline 对比（with/without-skill） | 需要量化改进 |
| QA checklist（prompt 版验证器） | 失败可枚举但不可代码检测（主观品味） |
| 无评估 | 探索性、失败成本低 |

## 4. 设计维度自检 checklist

设计自己的 skill 时逐项自问（观察 present/absent + 证据，不是打分）：

**指令层（SKILL.md 本身）**
- description 覆盖 what + when + 关键词？
- 核心指令明确，无歧义/矛盾？
- 有 worked examples？覆盖边界？
- 输出格式定义（schema/模板）？
- 错误/异常行为定义？
- 约束用"为什么"解释还是 bare MUST/NEVER？

**工具层（scripts/）**
- 工具覆盖度、重叠/差异？
- 实现质量（错误处理、并发/缓存）？
- 可复用 vs 过度耦合？

**策略层（prompt 工程）**
- 推理策略（CoT/分步/自纠错）？
- 角色设定？
- 上下文管理（长文本分块、渐进披露）？

**性能层（静态可推断 / 需运行）**
- 速度迹象（并行/缓存）——实际速度需运行
- 准确度迹象（few-shot/二次验证/schema）
- 鲁棒性迹象（重试/降级/异常捕获）
- 资源消耗（SKILL.md 大小、引用数）

## 5. 模式目录

可复用模式，按类分组。每条带适用条件——不带适用条件的是假通用。

### 触发面（trigger）
- **强势 description 对抗未充分触发**：嵌入关键词列表 + 何时用/不用。适用：依赖模型自动触发。不适用：user-invocable-only。
- **负面边界明确**：写清"何时不用"。适用：触发条件与别的技能/工具重叠。
- **调用轴划分**：disable-model-invocation 划出只许人触发的。适用：有些技能不该自动触发。不适用：harness 不支持该字段。

### 渐进式披露（disclosure）
- **知识形态决定披露策略**：见 §3.1。
- **重复三次的活固化成脚本/数据**：跨运行/技能反复发明的逻辑固化。适用：重复劳动明显。
- **按内容稳定性分层**：稳定规则内联，易变数据进库/引用。

### 可靠性约束（enforcement）
- **确定性规则机械拦截**：可代码检测的反模式用规则引擎。适用：失败可代码检测。不适用：主观品味。
- **约束式 prompt（规定不能怎么写）**：红线锁下限、放开上限。适用：要风格 + 反 AI 腔。不适用：需客观正确。
- **强约束 + 唯一创意通道**：锁死除一个创造性变量外的一切。适用：有明确 AI 味失败模式。
- **QA checklist 作 prompt 版验证器**：主观领域用 checklist + 失败信号 + 迭代修复。适用：失败不可代码检测。

### 跨平台分发（distribution）
- **模板生成跨平台**：单源 + 平台配置 + 模板引擎。适用：平台多且差异结构化。
- **单源+镜像同步是脆弱点**：多份拷贝必须配 CI 校验。

### 技能组织（organization）
- **强制引导 bootstrap**：session-start hook 强制加载引导技能。适用：技能不会被主动用。
- **路由器/元技能入口**：技能数增长后加入口分发或 checklist 兜底。适用：顶层技能 ≥ 5-7。

### 设计哲学（philosophy）
- **规范极简主义**：标准只规定最小契约，扩展塞 metadata。适用：写标准。不适用：写产品技能。
- **格式 vs 质量分层**：硬约束机器验，软约束人读指南。
- **人格扮演统一声音**：开头"channeling X"让输出有连贯声音。适用：方法论交付。

### 评估测试（evaluation）
- **技能 TDD（失败测试先行）**：先写失败场景再写技能。适用：质量必须可验证。
- **删除实验作可测试判断标准**：去掉 X 还成立→X 太装饰。适用：有"X 必须承担核心作用"的要求。
- **显式禁止复刻**：列旧模式逼重新发明。适用：示例驱动、模型易抄示例。

## 6. 常见陷阱（反模式）

1. **把建议当强制**：如把"500 行建议"升级成"超 200 行反模式"强制检查项。正确：区分 Must/Recommended。
2. **凭空发明通用架构**：如规定"plan/build/review/meta"默认架构——真实仓库无一使用。正确：组织方式因目标而异，无统一默认。
3. **规定 skill 数量上限**：如"3-4 个不超过 7"——多数成功仓库远超。正确：数量增长靠路由器，不靠硬上限。
4. **规范与工程规范混淆**：规范是跨实现最小契约，不是团队工程规范。正确：标准极小，团队规范另行。
5. **单源+镜像不同步**：多份拷贝不配 CI 校验必漂移。正确：零拷贝或配 sync 校验。
6. **frontmatter 堆字段没想清**：为"丰富"而堆字段是噪声。正确：字段匹配你的不信任点。
7. **可靠性手段错配**：用代码拦主观品味（过死）/ 用 prose 劝客观通病（拦不住）。正确：失败可代码检测→机械；主观→说服。

## 7. 设计流程

1. **诊断**：agent 出错主要在哪？（理解力/知识/纪律/审美）
2. **定知识形态**：能表格化？稳定还是易变？需脚本？
3. **定触发**：引导/描述/手动/路由？
4. **定可靠性**：说服/机械/混合？
5. **定披露结构**：小入口+refs/大自包含/脚本化/sections？
6. **定 frontmatter**：极简/丰富/调用轴？
7. **定分发**：单渠道/镜像/代码生成？
8. **定评估**：TDD/evals/QA checklist/无？
9. **自检**：跑 §4 的 4 层 × 6 维 checklist。
10. **落地 + 对抗评审**：写 SKILL.md + 引用；做对抗式自查（事实准确性 / 第一性原理深度 / 费曼清晰度 / 横向对照四维度），最好由没写过这份技能的视角评审。
