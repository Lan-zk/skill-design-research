# Skill 设计哲学

> 写给"要设计或开发 agent skill"的人的参考手册——告诉你有哪些标准要遵守、有哪些方案可选、具体怎么落地。内容是从多个开源 skill 仓库的实践中提炼出来的，但本文档可以独立阅读，不依赖任何项目背景。
>
> 用法：先读 §2，想清楚你信不过 agent 的到底是哪一面 → §3 据此选型 → §5 查可复用模式 → §7 走落地流程 → §4 自检 + §6 避坑。

## 1. 基础契约（Agent Skills 规范）

Agent Skills 是 Anthropic 牵头发起的开放标准（agentskills.io），规定了一个 skill 怎样打包才能被 agent 加载。

### 最小契约

一个 skill 就是一个目录，里面放一个 `SKILL.md`（由 YAML frontmatter + Markdown 正文组成），目录名就是技能名。目录里还可以放 `scripts/`（可执行代码）、`references/`（补充文档）、`assets/`（模板等资源），都是可选的。

### frontmatter 字段

frontmatter 是 `SKILL.md` 顶部那段 YAML，用来声明技能的元信息。规范只认这 6 个字段：

| 字段 | 必填 | 约束 |
|---|---|---|
| `name` | 是 | 1-64 字符；只能用小写字母/数字/连字符；连字符不能放开头、结尾或连续两个；**必须和父目录名一致** |
| `description` | 是 | 1-1024 字符，非空；应当写清"做什么 + 何时用"，并带上触发关键词 |
| `license` | 否 | 许可证名称，或指向 LICENSE 文件 |
| `compatibility` | 否 | ≤500 字符；说明运行环境需求 |
| `metadata` | 否 | string→string 的任意键值对（这是放扩展属性的合法位置） |
| `allowed-tools` | 否 | 空格分隔的"预批准工具"列表（实验性功能） |

只有这 6 个是规范允许的顶层字段；多写的字段会被验证器标成 "Unexpected fields"——如果你想加 provider 自定义的扩展属性，应该塞进 `metadata`。

### Must / Recommended / Convention 三分

规范里的要求分三档，别混在一起：

| 类别 | 含义 | 例子 |
|---|---|---|
| **Must** | 硬性规定，验证器会强制检查 | 必须有目录 + SKILL.md；name/description 必填且符合字符规则；只认 6 个字段 |
| **Recommended** | 只是建议，验证器不查 | 建议不超过 500 行/5000 token；建议做三层渐进式披露（元信息→正文→引用文件，分三层按需加载）；description 建议带关键词 |
| **Convention** | 业界普遍这么干但规范没强制（de facto 约定） | 技能放 `.agents/skills/` 目录；项目级技能覆盖用户级；用 `disable-model-invocation` 当过滤开关 |

**两条最容易误读的边界**：
- "500 行"是**建议，不是硬规则**。不少生产级 skill 超过 500 行仍然合理——信息密度高的时候，内容直接写在 SKILL.md 里（自包含）比拆成一堆引用文件更好。
- `disable-model-invocation` 虽然被实现指南承认为"过滤开关"，但**它没进规范的字段白名单**——属于 provider 扩展，换一个客户端可能就不认，跨客户端兼容性会受影响。

## 2. 第一性原理：先诊断，再定对策

设计 skill 之前，最该想清楚的不是"用什么模式"，而是**"你觉得 agent 出错，主要是卡在哪一类能力上"**。你信不过 agent 的哪一面（下文叫"不信任点"），就决定了该用哪种设计手段（下文叫“对策”）。这里没有"先进 vs 落后"之分，只有"匹不匹配你的诊断"。

### 四类不信任 → 四种对策

agent 的短板可以归成四类。下表四列对应：你不信任它的什么 → 你该怎么补 → 别人怎么做的 → 这种补法什么时候会失灵。

| 不信任点（信不过 agent 的…） | 对策（怎么补） | 代表做法 | 失效条件 |
|---|---|---|---|
| **理解力**（听不懂指令背后的道理） | 给它讲清"为什么"，而不是堆规则 | frontmatter 尽量精简，靠自然语言把"为什么这样做"讲透；脚本封装成黑盒直接调 | 模型理解力太弱时，你的"解释"对它反而是噪声；在不可信环境里"不限工具"会留安全漏洞 |
| **知识**（缺相关知识，会用过时或编造的内容） | 把知识做成文件让它查 | 维护一份"对抗过时知识"的清单；用 CSV + 检索引擎；把官方规范原样镜像进来 | 知识变动频繁时维护成本很高；隐式场景（模型自己都没想到要去查）触发不到 |
| **纪律**（懂但偷懒不照做） | 用流程和红线逼它就范 | session-start hook 强制加载引导；列一张"红旗清单"堵住常见的偷懒借口；用路由器分流 | 引导只是软约束，弱模型照样拦不住；用户一施压它就放弃原则 |
| **审美**（做了但品味/风格不对） | 用代码硬性拦截 | 确定性规则引擎；版式验证器脚本；用"红线列表"锁住下限 | 主观品味代码很难检测准；约束太死会把风格也磨掉 |

多数 skill 以一类短板为主，同时沾一点其他类。对策跟着诊断走。

**用四个问题给自己定位**——你的 agent 出错，主要是：
- "它不理解为什么要这么做" → 理解力问题 → 用解释补
- "它根本不知道这事" → 知识问题 → 把知识外化成文件
- "它知道但不照做" → 纪律问题 → 用流程逼
- "它做了但品味/风格跑偏" → 审美问题 → 用代码硬拦

## 3. 设计决策

下面每个决策都给出选项和"何时该选"（when-to-apply）。选的不是"最先进"的，而是"最匹配你的诊断"的。

### 3.1 知识形态 → 披露策略

你 skill 里的知识是哪种形态，决定了它该怎么被加载（"披露"就是"什么时候让模型看到多少内容"）：

| 知识形态 | 披露策略 | 何时该选 |
|---|---|---|
| 散文式原则（方法论、判断标准） | 单文件自包含 | 内容已成型稳定、没法表格化 |
| 结构化数据（如"产品类型→配置"） | 做成数据库 + 检索引擎 | 可枚举、易变、需要查询 |
| 确定性规则（反模式、版式要求） | 用规则引擎/验证器 | 是能被代码检测的通病 |
| 官方规范（格式、API） | 把规范原样镜像进技能 | 有权威外部源头 |

判断时问自己三个问题：能表格化吗？是稳定规则还是易变数据？需要脚本执行吗？

### 3.2 触发方式

技能怎么被唤起，有几种方式：

| 方式 | 何时该选 |
|---|---|
| 强制引导（session-start hook，会话一开始就加载） | 技能不会被模型主动调用，除非你强制加载；且你的运行环境（harness）支持 hook |
| description 匹配（默认机制） | 依赖模型自己判断该不该触发 |
| 收窄模型自主权（设 disable-model-invocation） | 有些技能绝不该被模型自动触发，只能让人手动调 |
| 路由器分发（一个入口技能转发给子技能） | 顶层技能数量 ≥ 5-7 个 |

模型有个通病——"该触发的时候没触发"（这叫"未充分触发"）。补偿办法分三档，从轻到重：写"强势 description"（枚举关键词 + 写明何时用/何时不用）、加路由器、强制引导。

### 3.3 可靠性执行：靠说服，还是靠机械拦截

agent 不照办时，你靠"讲道理"还是"写代码拦"，取决于它为什么不照办：

| 派别 | 手段 | 何时该选 |
|---|---|---|
| 说服派 | 自然语言原则（prose）、红线列表 | 失败原因是"模型没意识到这么做不对" |
| 机械派 | 代码检测、验证器、CI | 失败原因是"知道但管不住手"，且这种失败能被代码检测出来 |
| 混合派 | 用规则拦客观问题 + 用 LLM 批判兜底主观问题 | 客观失败和主观失败同时存在 |

这里"客观问题"指能明确判对错的事（比如违反格式），"主观问题"指品味、风格这类没有标准答案的事。

### 3.4 渐进式披露结构

"渐进式披露"指：入口文件尽量小，详细内容按需加载。具体怎么拆，看内容性质：

| 结构 | 何时该选 |
|---|---|
| 小入口 + references/（主文件小，细节放引用文件） | 内容量大且偏参考性 |
| 大自包含（全写在一个文件里） | 信息密度高、跳到别的文件反而更费劲 |
| 脚本化数据库（核心是检索） | 技能的核心价值就是查数据 |
| sections/manifest 分节按需加载（把工作流切成 section，用到哪段读哪段） | 超长工作流，每次只用其中一段 |
| 强制读引用（子命令调用的引用文件必读） | 跳过引用就会产出通用、不走心的输出 |

到底遵守"500 行建议"还是突破它？判据：信息密度高、且大部分任务都需要大部分内容时，自包含更优；否则就拆成引用。

### 3.5 frontmatter 哲学

frontmatter 该写多还是写少，反映你对模型的态度：

| 取向 | 何时该选 |
|---|---|
| 极简（只写 name + description） | 你信任模型的理解力 |
| 丰富（加 allowed-tools/version/hooks 等） | 想收敛工具面、分级加载 |
| 按调用方式划分（disable-model-invocation / user-invocable） | 想区分"谁触发"——模型还是人 |

注意：`disable-model-invocation`、`user-invocable`、`version`、`argument-hint` 这些字段都不在规范白名单里——它们是 provider 扩展，跨客户端兼容性会受影响。

### 3.6 跨平台分发

同一个技能要分发给多个 agent 平台时：

| 方式 | 何时该选 |
|---|---|
| 不放集成文件，只靠规范 | 只走单一标准渠道 |
| 少量手写 | 只覆盖 1-3 个平台 |
| 手写镜像（用占位符替换差异） | 平台多、但差异很小 |
| 模板引擎生成 | 平台多（≥5）、且差异是结构化的 |
| 代码生成 host 配置 | 平台多、且需要 host 特定逻辑 |

**"单源 + 镜像"的同步是固有脆弱点**：只要同一份内容有多份拷贝，就必须配上 CI/sync 校验，否则各拷贝迟早会漂移得不一致。零拷贝（单文件即唯一来源）最稳，但会锁死在单平台上。

### 3.7 评估与测试

怎么验证技能真有用：

| 方式 | 何时该选 |
|---|---|
| TDD 压力测试先行（先写失败场景再写技能） | 失败成本高、你有工程能力 |
| evals baseline 对比（带技能 vs 不带技能） | 需要量化"加了这个技能改进了多少" |
| QA checklist（prompt 版的验证器） | 失败能列出来但代码检测不了（主观品味） |
| 不做评估 | 探索性、失败成本低 |

## 4. 设计维度自检 checklist

设计自己的 skill 时，按下面四层逐项自问。对每一项判断"有没有"并举出证据，不是打分：

**指令层（SKILL.md 本身）**
- description 覆盖了"做什么 + 何时用 + 关键词"吗？
- 核心指令明确、无歧义、无自相矛盾吗？
- 有可参照的实例（worked examples）吗？覆盖了边界情况吗？
- 输出格式定义清楚了吗（schema/模板）？
- 错误和异常行为定义了吗？
- 约束是用"为什么"解释的，还是只甩出 MUST/NEVER？

**工具层（scripts/）**
- 工具覆盖度如何？有没有重叠或差异？
- 实现质量如何（错误处理、并发/缓存）？
- 是可复用的，还是过度耦合？

**策略层（prompt 工程）**
- 推理策略（CoT/分步/自纠错）？
- 角色设定？
- 上下文管理（长文本分块、渐进披露）？

**性能层（部分可静态推断，部分需实际运行）**
- 速度迹象（并行/缓存）——实际速度得运行才知道
- 准确度迹象（few-shot/二次验证/schema）
- 鲁棒性迹象（重试/降级/异常捕获）
- 资源消耗（SKILL.md 大小、引用数）

## 5. 模式目录

下面是可复用模式，按类分组。每条都附适用条件——不写适用条件的"通用模式"其实是假通用（没有放之四海皆准的模式）。

### 触发面（trigger）
- **强势 description 对抗"未充分触发"**：在 description 里嵌入关键词列表 + 写明何时用/何时不用。适用：依赖模型自动触发。不适用：技能只允许人手动调用。
- **负面边界明确**：写清"何时不用"。适用：触发条件跟别的技能/工具有重叠。
- **按调用方式划分**：用 disable-model-invocation 划出"只许人触发"的技能。适用：有些技能不该被自动触发。不适用：harness 不支持这个字段。

### 渐进式披露（disclosure）
- **知识形态决定披露策略**：见 §3.1。
- **重复出现三次的工作，就固化成脚本/数据**：跨运行、跨技能反复发明的逻辑，固化下来。适用：重复劳动明显。
- **按内容稳定性分层**：稳定规则内联在主文件，易变数据放进库/引用文件。

### 可靠性约束（enforcement）
- **确定性规则机械拦截**：可代码检测的反模式，用规则引擎挡住。适用：失败能被代码检测。不适用：主观品味。
- **约束式 prompt（规定"不能怎么写"）**：用红线锁住下限、放开上限。适用：要风格、要反 AI 腔。不适用：需要客观正确。
- **强约束 + 唯一创意通道**：把除一个创造性维度之外的一切都锁死，只留一个口子让模型发挥（比如锁死结构和篇幅，只放开措辞）。适用：有明确的"AI 味"失败模式。
- **QA checklist 当 prompt 里的验证器**：在主观领域，用一份 checklist + 失败信号列表，让模型迭代修复自己。适用：失败不可代码检测。

### 跨平台分发（distribution）
- **模板生成跨平台**：单源 + 平台配置 + 模板引擎。适用：平台多且差异是结构化的。
- **单源 + 镜像同步是脆弱点**：多份拷贝必须配 CI 校验。

### 技能组织（organization）
- **强制引导 bootstrap**：用 session-start hook 在会话一开始就强制加载引导技能。适用：技能不会被主动用。
- **路由器/元技能入口**：技能数变多后，加一个入口技能做分发，或用 checklist 兜底。适用：顶层技能 ≥ 5-7。

### 设计哲学（philosophy）
- **规范极简主义**：标准只规定最小契约，扩展属性塞进 metadata。适用：你在写标准。不适用：你在写产品技能。
- **格式 vs 质量分层**：硬约束（格式类）交给机器验证，软约束（质量类）靠人读指南。
- **人格扮演统一声音**：开头以"channeling X"（化身某个人）的口吻写，让输出有连贯的声音。适用：交付的是一套方法论。

### 评估测试（evaluation）
- **技能 TDD（失败测试先行）**：先写"会失败的场景"再写技能。适用：质量必须可验证。
- **用"删除实验"作可测试的判断标准**：把某个成分 X 去掉后技能还成立 → 说明 X 太装饰、可有可无。适用：有"X 必须承担核心作用"的要求。
- **显式禁止复刻**：把要避免的旧模式列出来，逼模型重新发明而非照抄。适用：示例驱动、模型容易照抄示例。

## 6. 常见陷阱（反模式）

1. **把建议当强制**：比如把"500 行建议"升级成"超 200 行就是反模式"的强制检查项。正确做法：分清 Must 和 Recommended。
2. **凭空发明"通用架构"**：比如规定技能都得按"plan/build/review/meta"分层——真实仓库里没有一个这么干。正确做法：组织方式因目标而异，没有统一默认。
3. **规定 skill 数量上限**：比如"3-4 个，不超过 7"——多数成功仓库远超这个数。正确做法：数量增长靠路由器消化，不靠硬上限。
4. **规范与工程规范混淆**：规范是跨实现的最小契约，不是团队内部的工程规范。正确做法：标准保持极小，团队规范另外定。
5. **单源 + 镜像不同步**：多份拷贝不配 CI 校验，必然漂移。正确做法：零拷贝，或配上 sync 校验。
6. **frontmatter 堆字段但没想清为什么**：为了显得"丰富"而堆字段，其实是噪声。正确做法：字段要匹配你的不信任点。
7. **可靠性手段错配**：用代码拦主观品味（过死），或用自然语言劝客观通病（拦不住）。正确做法：失败能代码检测→用机械手段；主观问题→用说服。

## 7. 设计流程

1. **诊断**：agent 出错主要卡在哪一类能力？（理解力/知识/纪律/审美）
2. **定知识形态**：能表格化吗？稳定还是易变？需要脚本吗？
3. **定触发**：强制引导 / description 匹配 / 手动 / 路由器？
4. **定可靠性**：说服 / 机械 / 混合？
5. **定披露结构**：小入口+references / 大自包含 / 脚本化 / sections？
6. **定 frontmatter**：极简 / 丰富 / 按调用方式划分？
7. **定分发**：单渠道 / 镜像 / 代码生成？
8. **定评估**：TDD / evals / QA checklist / 不做？
9. **自检**：跑一遍 §4 的四层 checklist。
10. **落地 + 对抗评审**：写 SKILL.md + 引用文件；然后做对抗式自查（四个维度：事实准确性 / 第一性原理深度 / 费曼清晰度 / 横向对照），最好找一个没写过这份技能的人来评审。
